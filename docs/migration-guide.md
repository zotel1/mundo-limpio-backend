# Guía de Migración: MySQL → PostgreSQL

> **WHAT**: Procedimiento completo para migrar la base de datos de MySQL 8.0 a PostgreSQL 16
> **WHY**: La migración habilita el deployment en Supabase y mejora el rendimiento de consultas con tipos nativos
> **DIFFERENCES**: Esta guía asume migración in-place en el mismo servidor; para migración a otro host, ajustar las variables de conexión

## 1. Pre-Cutover Checklist

### Día anterior al cutover

- [ ] **Backup completo de MySQL**
  ```bash
  mysqldump --single-transaction --routines --triggers \
    -h $MYSQL_HOST -u $MYSQL_USER -p$MYSQL_PASSWORD \
    $MYSQL_DATABASE > backup_pre_migration_$(date +%Y%m%d).sql
  ```
  - Verificar que el dump no esté vacío: `wc -l backup_pre_migration_*.sql`
  - Guardar el backup en al menos 2 ubicaciones (local + cloud storage)

- [ ] **Verificar espacio en disco**
  - El dump MySQL + archivos transformados ocupan ~2-3× el tamaño de la base de datos
  - Asegurar al menos 5GB libres para bases de datos de hasta 1GB

- [ ] **Verificar versión de PostgreSQL**
  ```bash
  psql --version  # Debe ser >= 16.0
  ```
  - Si usás Docker: `docker run --rm postgres:16 psql --version`

- [ ] **Verificar conectividad a PostgreSQL destino**
  ```bash
  PGPASSWORD=$PG_PASSWORD psql -h $PG_HOST -p $PG_PORT -U $PG_USER -d $PG_DATABASE -c "SELECT version();"
  ```

- [ ] **Preparar PostgreSQL destino (base de datos vacía)**
  ```sql
  -- Conectarse como superuser y crear la base de datos si no existe
  CREATE DATABASE mundolimpio WITH ENCODING 'UTF8' LC_COLLATE='es_AR.UTF-8' LC_CTYPE='es_AR.UTF-8';
  ```
  - Para Railway/Supabase: la base de datos ya existe; verificar encoding UTF8

- [ ] **Ejecutar dry-run del script de migración** (ver [Sección 5](#5-script-dry-run))
  - Confirma que las transformaciones sed producen SQL válido
  - Verifica que los conteos de filas de MySQL son razonables
  - Identifica warnings o errores potenciales ANTES del cutover real

- [ ] **Notificar al equipo**
  - Comunicar la ventana de mantenimiento planificada
  - Asegurar que no haya deploys programados durante la ventana

- [ ] **Preparar el nuevo JAR de la aplicación**
  ```bash
  ./mvnw clean package -DskipTests
  ```
  - Verificar que `application-prod.yml` apunte a PostgreSQL
  - El JAR compilado incluye las dependencias de PostgreSQL y Flyway para PG

- [ ] **Verificar variables de entorno para el script**
  ```bash
  # Asegurar que estas variables estén definidas (o usar defaults)
  export MYSQL_HOST=localhost
  export MYSQL_PORT=3306
  export MYSQL_USER=root
  export MYSQL_PASSWORD=tu_password
  export MYSQL_DATABASE=mundolimpio

  export PG_HOST=localhost
  export PG_PORT=5432
  export PG_USER=postgres
  export PG_PASSWORD=tu_password
  export PG_DATABASE=mundolimpio
  ```

### Inmediatamente antes del cutover

- [ ] **Detener la aplicación** (para evitar escrituras durante la migración)
  ```bash
  # Docker
  docker-compose down backend_api
  # O si corre directamente
  pkill -f 'java.*mundolimpio'
  ```

- [ ] **Ejecutar dump final de MySQL** (captura el último estado)
  - Usar el script de migración que incluye el dump como primer paso

## 2. Procedimiento de Cutover

### Estimación de tiempos

| Paso | Tiempo estimado | Descripción |
|------|-----------------|-------------|
| 1. Dump MySQL | 1-5 min | Depende del tamaño de la DB (aproximadamente 1 min por cada 100MB) |
| 2. Transformación sed | <30 seg | Procesamiento local de archivo de texto, no depende del tamaño de DB |
| 3. Import psql | 2-10 min | Más lento que el dump; ~2-3 min por cada 100MB |
| 4. Verificación COUNT(*) | 1-3 min | Depende de cantidad de tablas y volumen de datos |
| 5. Deploy nueva app | 1-2 min | Compilada previamente |
| 6. Smoke tests | 2-5 min | Manual o automatizado |
| **Total ventana recomendada** | **15-30 min** | Con margen para imprevistos |

> **NOTA**: Para bases de datos >1GB, duplicar estas estimaciones. El dump y el import son los pasos que escalan linealmente con el tamaño.

### Paso a paso

#### Paso 1: Ejecutar el script de migración

```bash
cd /ruta/a/mundo-limpio-backend
chmod +x scripts/migrate-db.sh
./scripts/migrate-db.sh
```

El script ejecuta automáticamente las 4 fases:

**Fase 1 — Export MySQL**: `mysqldump --compatible=postgresql` exporta esquema y datos.

**Fase 2 — Transformar**: Aplica 12 reglas `sed` para convertir sintaxis MySQL a PostgreSQL:
- `BIGINT AUTO_INCREMENT` → `BIGINT GENERATED BY DEFAULT AS IDENTITY`
- `TINYINT(1)` → `BOOLEAN`
- `ENUM('a','b')` → `VARCHAR(255) CHECK (col IN ('a','b'))`
- Remueve `ENGINE=InnoDB`
- `utf8mb4` → `UTF8`
- Remueve comentarios versionados `/*!40101 ... */`
- Remueve `SET sql_mode`, `SET time_zone`, etc.
- Normaliza comentarios de línea (`--text` → `-- text`)
- Remueve `AUTO_INCREMENT=N` de CREATE TABLE
- Remueve `COLLATE` específico de columna
- Convierte `b'1'`/`b'0'` → `TRUE`/`FALSE`
- Transforma `USE database` a comentario

**Fase 3 — Import PostgreSQL**: `psql` importa el SQL transformado en una transacción atómica (`--single-transaction`).

**Fase 4 — Verificar**: Compara `SELECT COUNT(*)` para cada tabla entre MySQL y PostgreSQL.

#### Paso 2: Verificar la salida

Al finalizar, el script muestra una tabla comparativa:

```
TABLA                                MYSQL COUNT(*)  POSTGRES COUNT(*)    MATCH
----------------------------------- --------------- --------------- ----------
bulk_products                                     3               3       ✅
inventory                                         5               5       ✅
inventory_adjustments                            12              12       ✅
products                                          8               8       ✅
production_batches                               15              15       ✅
sale_items                                       20              20       ✅
sales                                             7               7       ✅
```

**SI TODAS LAS FILAS MUESTRAN ✅**: Continuar al Paso 3.

**SI ALGUNA FILA MUESTRA ❌**: Detener. Revisar el log del script para identificar la tabla con discrepancia. No continuar con el deploy.

#### Paso 3: Ejecutar migraciones Flyway

El script de migración importa los datos existentes, pero NO ejecuta Flyway. Las migraciones Flyway se ejecutan automáticamente al iniciar la aplicación Spring Boot.

```bash
# Verificar que Flyway migrará correctamente (modo validate, sin aplicar)
java -jar target/mundolimpio-0.0.1-SNAPSHOT.jar \
  --spring.flyway.enabled=true \
  --spring.flyway.validate-on-migrate=true
```

Flyway ejecutará V1-V4 contra PostgreSQL limpio. Como la base de datos ya tiene los datos migrados, Flyway solo verificará que el esquema coincida (modo `validate` con `ddl-auto: validate`).

#### Paso 4: Deploy de la nueva aplicación

```bash
# Con Docker
docker-compose up -d backend_api

# O directamente
java -jar target/mundolimpio-0.0.1-SNAPSHOT.jar --spring.profiles.active=prod
```

#### Paso 5: Ejecutar smoke tests

Ver [Sección 3](#3-smoke-test-checklist).

## 3. Smoke Test Checklist

Ejecutar estas verificaciones INMEDIATAMENTE después del deploy para confirmar que la migración fue exitosa.

### 3.1 Health Check

- [ ] **Endpoint de health**
  ```bash
  curl -s http://localhost:8080/actuator/health | jq .
  ```
  Esperado: `{"status": "UP"}`

- [ ] **Swagger UI**
  ```bash
  curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/swagger-ui.html
  ```
  Esperado: `200`

### 3.2 Autenticación (sin datos sensibles)

- [ ] **Login funciona contra PostgreSQL**
  ```bash
  curl -s -X POST http://localhost:8080/api/v1/auth/login \
    -H "Content-Type: application/json" \
    -d '{"username":"admin","password":"admin123"}' | jq .
  ```
  Esperado: Response con `accessToken` y `refreshToken`

- [ ] **Refresh token funciona**
  ```bash
  # Usar el refreshToken del paso anterior
  curl -s -X POST http://localhost:8080/api/v1/auth/refresh \
    -H "Content-Type: application/json" \
    -d '{"refreshToken":"...token..."}' | jq .
  ```
  Esperado: Nuevo `accessToken`

### 3.3 Endpoints CRUD (con token JWT)

Reemplazar `$TOKEN` con el accessToken obtenido en 3.2.

- [ ] **Listar productos**
  ```bash
  curl -s http://localhost:8080/api/v1/products \
    -H "Authorization: Bearer $TOKEN" | jq '. | length'
  ```
  Esperado: Número de productos > 0, igual al conteo verificado en Paso 2.

- [ ] **Listar materia prima**
  ```bash
  curl -s http://localhost:8080/api/v1/bulk-products \
    -H "Authorization: Bearer $TOKEN" | jq '. | length'
  ```
  Esperado: Número de bulk_products > 0.

- [ ] **Listar lotes de producción**
  ```bash
  curl -s http://localhost:8080/api/v1/production-batches/product/1 \
    -H "Authorization: Bearer $TOKEN" | jq '. | length'
  ```
  Esperado: Lista de lotes (puede ser vacía si el producto no tiene lotes).

- [ ] **Consultar inventario**
  ```bash
  curl -s http://localhost:8080/api/v1/inventory/1 \
    -H "Authorization: Bearer $TOKEN" | jq .
  ```
  Esperado: Objeto con `currentStock` y `minStockThreshold`.

- [ ] **Crear venta (FIFO)**
  ```bash
  curl -s -X POST http://localhost:8080/api/v1/sales \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"productId":1,"quantity":1}' | jq .
  ```
  Esperado: Response 200/201 con `totalAmount` y `items`.

### 3.4 Integridad de Datos

- [ ] **Verificar IDs secuenciales**: Crear un nuevo producto y verificar que el ID se asigna correctamente (IDENTITY).
  ```bash
  curl -s -X POST http://localhost:8080/api/v1/products \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"sku":"SMOKE-TEST-001","name":"Producto Smoke Test","minPrice":9.99}' | jq '.id'
  ```
  Esperado: ID numérico (no nulo, > 0).

  Luego eliminar el producto de prueba:
  ```bash
  curl -s -X DELETE http://localhost:8080/api/v1/products/<ID> \
    -H "Authorization: Bearer $TOKEN"
  ```

- [ ] **Verificar stock post-FIFO**: Después de crear una venta, verificar que el stock se descontó correctamente.
  ```bash
  curl -s http://localhost:8080/api/v1/inventory/<productId> \
    -H "Authorization: Bearer $TOKEN" | jq '.currentStock'
  ```
  Esperado: Stock anterior menos la cantidad vendida.

- [ ] **Verificar versión optimistic locking**: La columna `version` debe ser BIGINT >= 0 en todas las entidades con `@Version`.

- [ ] **Verificar encoding UTF-8**: Crear un producto con caracteres especiales (ñ, tildes) y recuperarlo.
  ```bash
  curl -s -X POST http://localhost:8080/api/v1/products \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"sku":"CAFÉ-001","name":"Café con leche","minPrice":15.50}' | jq '.name'
  ```
  Esperado: `"Café con leche"` (caracteres preservados).

### 3.5 Logs

- [ ] **Revisar logs de la aplicación**
  ```bash
  docker-compose logs backend_api | grep -i "error\|exception\|postgresql\|flyway"
  ```
  Esperado: Sin errores. Flyway debe mostrar "Successfully applied N migrations".

## 4. Procedimiento de Rollback

> **WHAT**: Procedimiento para revertir a MySQL si la migración a PostgreSQL falla en producción
> **WHY**: Una ventana de rollback documentada reduce el riesgo de downtime prolongado
> **DIFFERENCES**: El rollback revierte base de datos, dependencias y configuración en orden inverso al cutover

### 4.1 Cuándo hacer rollback

Ejecutar rollback si CUALQUIERA de estas condiciones se cumple:
- ❌ La verificación de `COUNT(*)` muestra discrepancias en una o más tablas
- ❌ Flyway falla al aplicar migraciones (error de sintaxis o checksum)
- ❌ La aplicación no inicia (error de conexión a PostgreSQL, dialect mismatch)
- ❌ Los smoke tests fallan (endpoints devuelven 500, datos corruptos)
- ❌ La ventana de mantenimiento se excede sin haber completado todos los pasos

### 4.2 Procedimiento paso a paso

**Tiempo estimado**: 10-20 minutos (depende del tamaño del restore de MySQL)

#### Paso 1: Detener la nueva aplicación

```bash
docker-compose down backend_api
# O
pkill -f 'java.*mundolimpio'
```

#### Paso 2: Restaurar dump de MySQL

```bash
# WHAT: Restaura el backup pre-migración a MySQL
# WHY: Vuelve la base de datos al estado exacto anterior al cutover
mysql -h $MYSQL_HOST -P $MYSQL_PORT -u $MYSQL_USER -p$MYSQL_PASSWORD \
  $MYSQL_DATABASE < backup_pre_migration_YYYYMMDD.sql

# Verificar que el restore fue exitoso
mysql -h $MYSQL_HOST -u $MYSQL_USER -p$MYSQL_PASSWORD \
  -e "SELECT COUNT(*) FROM products;" $MYSQL_DATABASE
```

> **IMPORTANTE**: Si MySQL tiene datos posteriores al backup (porque la app siguió corriendo brevemente), primero hacer un DROP de la base de datos y recrearla antes del restore:
> ```sql
> DROP DATABASE IF EXISTS mundolimpio;
> CREATE DATABASE mundolimpio CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
> ```

#### Paso 3: Revertir `pom.xml` a dependencias MySQL

```bash
git checkout pom.xml
```

Esto revierte los cambios de PR #1.1 y #1.2:
- Restaura `mysql-connector-j` y `flyway-mysql`
- Remueve `postgresql` y `flyway-database-postgresql`
- Restaura `h2` como base de datos de tests
- Remueve `testcontainers:postgresql`

#### Paso 4: Revertir `docker-compose.yml` a MySQL

```bash
git checkout docker-compose.yml
```

Esto revierte:
- Servicio `postgres_db` → `mysql_db`
- Imagen `postgres:16-alpine` → `mysql:8.0`
- Variables de entorno POSTGRES_* → MYSQL_*
- Puerto 5432 → 3306
- Healthcheck `pg_isready` → `mysqladmin ping`
- `SPRING_DATASOURCE_URL` → `jdbc:mysql://...`

#### Paso 5: Revertir configuraciones de aplicación

```bash
git checkout src/main/resources/application-prod.yml
git checkout src/test/resources/application-test.yml
```

Esto revierte:
- `driver-class-name: org.postgresql.Driver` → `com.mysql.cj.jdbc.Driver`
- URL JDBC con `jdbc:postgresql://` → `jdbc:mysql://`
- `hibernate.dialect: PostgreSQLDialect` → MySQL (dialect auto-detect)

#### Paso 6: Restaurar migraciones Flyway MySQL originales

```bash
# WHAT: Copia los archivos .bak (MySQL original) sobre los archivos PostgreSQL
# WHY: Flyway necesita el DDL MySQL original para validar checksums en MySQL
cp src/main/resources/db/migration/V1__Initial_Schema.sql.bak \
   src/main/resources/db/migration/V1__Initial_Schema.sql

cp src/main/resources/db/migration/V2__Create_Inventory_Tables.sql.bak \
   src/main/resources/db/migration/V2__Create_Inventory_Tables.sql
```

> **NOTA**: V3 y V4 no requieren rollback porque su DDL es cross-dialect (solo ALTER TABLE, sin sintaxis específica de motor).

#### Paso 7: Recompilar y deployar la aplicación original

```bash
# Recompilar con dependencias MySQL
./mvnw clean package -DskipTests

# Deployar
docker-compose up -d backend_api
```

#### Paso 8: Verificar rollback exitoso

Ejecutar el smoke test checklist ([Sección 3](#3-smoke-test-checklist)) pero apuntando a MySQL.

Verificaciones clave:
- [ ] Health check: `{"status": "UP"}`
- [ ] Login funciona: devuelve `accessToken`
- [ ] CRUD de productos: lista la misma cantidad de productos que antes del cutover
- [ ] Logs sin errores de conexión o dialecto
- [ ] Flyway muestra `Success` en todas las migraciones

### 4.3 Rollback solo de aplicación (sin revertir base de datos)

Si PostgreSQL funciona correctamente pero la aplicación tiene un bug NO relacionado con la migración, se puede hacer un rollback parcial:

```bash
# Solo revertir código de aplicación, mantener PostgreSQL
git checkout pom.xml docker-compose.yml src/main/resources/application-prod.yml
# NO restaurar migraciones .bak
./mvnw clean package -DskipTests
docker-compose up -d backend_api
```

## 5. Script Dry-Run

> **WHAT**: El script `migrate-db.sh` soporta un modo `--dry-run` que simula la migración sin tocar PostgreSQL
> **WHY**: Permite validar el dump, las transformaciones y los conteos ANTES del cutover real
> **DIFFERENCES**: El dry-run exporta y transforma pero NO importa a PostgreSQL; ideal para validación pre-cutover

### 5.1 Ejecutar dry-run

export MYSQL_HOST=localhost
export MYSQL_PORT=3306
export MYSQL_USER=root
export MYSQL_PASSWORD=tu_password
export MYSQL_DATABASE=mundolimpio

# Ejecutar en modo dry-run
./scripts/migrate-db.sh --dry-run

### 5.3 Cómo validar el dry-run

#### 5.3.1 Revisar transformaciones

```bash
# Ver diferencias entre el dump original y el transformado
diff migration_dryrun_YYYYMMDD_HHMMSS.sql migration_dryrun_YYYYMMDD_HHMMSS.sql.transformed | head -100
```

Verificar que las diferencias incluyen (pero no están limitadas a):
- `AUTO_INCREMENT` reemplazado por `GENERATED BY DEFAULT AS IDENTITY`
- `ENGINE=InnoDB` removido
- `utf8mb4` reemplazado por `UTF8`
- `/*!40101 ... */` removido o transformado
- `SET sql_mode` removido
- `TINYINT(1)` reemplazado por `BOOLEAN` (si aplica)

#### 5.3.2 Verificar sintaxis PostgreSQL

```bash
# WHAT: Intenta parsear el SQL transformado con psql en modo --dry-run
# WHY: Detecta errores de sintaxis PostgreSQL ANTES del cutover real
# NOTA: psql acepta --set con un archivo pero no tiene modo dry-run nativo.
#       En su lugar, se puede importar a una base de datos temporal de prueba.
PGPASSWORD=test psql -h localhost -p 5432 -U postgres -d postgres \
  -c "CREATE DATABASE test_dryrun TEMPLATE template0 ENCODING 'UTF8';" 2>/dev/null

# Intentar importar (fallará si hay errores de sintaxis)
PGPASSWORD=test psql -h localhost -p 5432 -U postgres -d test_dryrun \
  --set ON_ERROR_STOP=1 \
  --file=migration_dryrun_YYYYMMDD_HHMMSS.sql.transformed

# Limpiar
PGPASSWORD=test psql -h localhost -p 5432 -U postgres -d postgres \
  -c "DROP DATABASE test_dryrun;"
```

#### 5.3.3 Validar conteos de filas

Los conteos mostrados al final del dry-run son tu REFERENCIA. Después de la migración real, el script ejecuta automáticamente la verificación de `COUNT(*)` comparando estos valores contra PostgreSQL.

**Regla de validación**: Si los conteos no coinciden en TODAS las tablas, la migración no es exitosa y se debe ejecutar el rollback.

#### 5.3.4 Validar IDs y secuencias

```bash
# WHAT: Verifica que los IDs AUTO_INCREMENT se preserven como IDENTITY
# WHY: Si los IDs cambian durante la migración, las FKs se rompen
grep -i "GENERATED BY DEFAULT AS IDENTITY" migration_dryrun_YYYYMMDD_HHMMSS.sql.transformed | wc -l
```

Comparar este número con:
```bash
grep -i "AUTO_INCREMENT" migration_dryrun_YYYYMMDD_HHMMSS.sql | wc -l
```

Ambos conteos DEBEN ser iguales (cada AUTO_INCREMENT fue convertido a IDENTITY).

### 5.4 Troubleshooting del dry-run

| Problema | Causa probable | Solución |
|----------|---------------|----------|
| `mysqldump: not found` | MySQL client no instalado | Instalar `mysql-client` o usar Docker: `docker run --rm mysql:8.0 mysqldump ...` |
| `mysqldump` produce un archivo vacío | Credenciales incorrectas o DB no existe | Verificar `MYSQL_HOST`, `MYSQL_USER`, `MYSQL_PASSWORD`, `MYSQL_DATABASE` |
| `sed` transformaciones no se aplican | Archivo ya fue transformado (idempotencia) | Si el dump ya es PG-compatible, las transformaciones no matchean (esperado) |
| Archivo transformado tiene sintaxis inválida | ENUM multilínea no capturado | El script maneja ENUMs de una línea; los multilínea requieren ajuste manual (poco común) |
| El conteo de IDENTITY no coincide con AUTO_INCREMENT | Alguna tabla usa `INT AUTO_INCREMENT` en vez de `BIGINT` | La transformación solo captura `BIGINT AUTO_INCREMENT`; `INT` debe ajustarse manualmente |

## 6. Configuración por Entorno

### 6.1 Variables de entorno requeridas

| Variable | Entorno | Default | Descripción |
|----------|---------|---------|-------------|
| `MYSQL_HOST` | Script migración | `localhost` | Host del MySQL origen |
| `MYSQL_PORT` | Script migración | `3306` | Puerto MySQL |
| `MYSQL_USER` | Script migración | `root` | Usuario MySQL con permisos de lectura |
| `MYSQL_PASSWORD` | Script migración | (vacío) | Password MySQL |
| `MYSQL_DATABASE` | Script migración | `mundolimpio` | Nombre de la DB MySQL |
| `PG_HOST` | Script + App | `localhost` | Host PostgreSQL destino |
| `PG_PORT` | Script + App | `5432` | Puerto PostgreSQL |
| `PG_USER` | Script + App | `postgres` | Usuario PostgreSQL |
| `PG_PASSWORD` | Script + App | (vacío) | Password PostgreSQL |
| `PG_DATABASE` | Script + App | `mundolimpio` | Nombre de la DB PostgreSQL |
| `SPRING_PROFILES_ACTIVE` | App | `prod` | Profile de Spring Boot |
| `SPRING_DATASOURCE_URL` | App (Docker) | `jdbc:postgresql://postgres_db:5432/${DB_NAME}` | URL JDBC completa |
| `SPRING_DATASOURCE_USERNAME` | App (Docker) | `${DB_USER}` | Usuario DB para Spring |
| `SPRING_DATASOURCE_PASSWORD` | App (Docker) | `${DB_PASSWORD}` | Password DB para Spring |

### 6.2 Railway / Supabase

Para deployments en Railway o Supabase, las variables de entorno son gestionadas por la plataforma:

- **Railway**: Configurar `PGHOST`, `PGPORT`, `PGDATABASE`, `PGUSER`, `PGPASSWORD` en el dashboard del servicio
- **Supabase**: Usar las credenciales del pooler de conexiones (port 6543 para transacciones, 5432 para sesiones)

### 6.3 Docker Compose local

El archivo `docker-compose.yml` ya está configurado para PostgreSQL. Variables en `.env`:

```env
DB_NAME=mundolimpio
DB_USER=postgres
DB_PASSWORD=postgres
DB_PORT=5432
```
